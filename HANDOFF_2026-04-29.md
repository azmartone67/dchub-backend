## 🟢 RESOLVED 2026-04-30 morning

**Root cause:** The dchub-backend Railway service was configured as a **cron job** (`Cron Schedule: 0 6 * * *`), not a continuous web service. Railway's `cronReady` status means "armed to fire at next scheduled time", not "currently running". Builds succeeded, but containers only spun up briefly at 6 AM UTC each day, then exited — explaining empty deploy logs and 502s on every endpoint between cron windows.

**Fix:** Removed the cron schedule in Railway → Settings → Cron Schedule → cleared. Service now runs continuously. Redeployed commit `6513b40` (Procfile cleanup) and observed full Flask startup logs streaming.

**How to spot this in the future:**
- If `cronReady` appears in environment status → cron-mode is on
- If Settings → Cron Schedule has any value → service won't run continuously
- Build success + empty deploy logs + persistent 502 = check cron config first

**Lesson:** Cron configuration was sitting in plain sight in yesterday's Settings paste. Always look for `Cron Schedule` value when a Railway service "builds but doesn't run".
