## Item 2 — Apply `/mcp/manifest` patch to the `dchub-mcp-server` repo

**Status now:** `/mcp/manifest` on dchub.cloud returns 404 because the
zone-level `4.8.5-mcp-landing` Workers Route proxies all `/mcp/*` to the
Express service at `dchub-mcp-server-production.up.railway.app`, which
doesn't have the manifest endpoint. We worked around it by adding
`/api/v1/mcp/manifest` (commit `ae81bfec`), but the canonical path stays
broken until the Express service ships a fix.

**Action:**

1. `cd` into your `dchub-mcp-server` repo (separate from `dchub-backend`).
2. Open `server.mjs`.
3. Paste the patch from
   [`dchub-mcp-v2.1/server.mjs` lines 549–593](dchub-mcp-v2.1/server.mjs)
   in this repo — it adds a cached `/mcp/manifest` handler that proxies
   to `${API_BASE}/mcp/manifest`. The block starts with
   `// ── Phase ZZZZZ-round3 (2026-05-23): /mcp/manifest passthrough` —
   insert it right before `app.listen(PORT, '0.0.0.0', ...)`.
4. Commit + push. Railway will redeploy `dchub-mcp-server` automatically.
5. Verify:
   ```bash
   curl -sI https://dchub.cloud/mcp/manifest | head -3
   curl -s  https://dchub.cloud/mcp/manifest | head -c 200
   ```
   Should now return 200 with the Flask manifest body.

The patch file also fixes a JavaScript syntax error at the top of the
file (lines 2–3 are Python-style `#` comments that won't parse if Node
re-evaluates the file). Apply both.

## Item 7 — Fix Twitter auto-publish (X API 403)

**Current Railway log:**
```
Twitter auto-publish failed for 79: X API error 403:
"When authenticating requests to the Twitter API v2 endpoints, you must
 use keys and tokens from a Twitter developer App that is attached to
 a Project."
```

**Action (one-time, ~5 min):**

1. Go to <https://developer.twitter.com/en/portal/projects-and-apps>.
2. Create a new Project (or move the existing DC Hub app into a Project
   if one exists). Free tier is fine.
3. Within the Project, attach the existing DC Hub app (the one whose
   keys are currently in Railway).
4. Regenerate the keys (Bearer token, API key+secret, access
   token+secret) — they invalidate on project attachment.
5. Update Railway env vars on the `dchub-backend` service:
   ```
   TWITTER_API_KEY
   TWITTER_API_SECRET
   TWITTER_ACCESS_TOKEN
   TWITTER_ACCESS_SECRET
   TWITTER_BEARER_TOKEN
   ```
6. Trigger a redeploy (`railway up` or push any commit).
7. Verify on the next auto-press cycle — within 6 hours you should see:
   ```
   Twitter auto-publish OK for <press-id>
   ```
   instead of the 403.

## Item 8 — Activate visitor-intel IPinfo enrichment

**Current state:** Module is wired (`routes/visitor_intelligence.py`,
commit `5b5e1ac1`), endpoint exists at `/api/v1/admin/ip-enrich`, but
returns `{"provider": "unconfigured"}` because no token is set. The
top-25 anon-IP section on `/visitor-intelligence` will start populating
the moment the token lands.

**Action (one-time, ~3 min):**

1. Sign up for IPinfo at <https://ipinfo.io/signup>. Paid plan starts
   at $99/mo for 250k lookups; free tier (50k/mo) is enough to validate.
2. Copy your access token from <https://ipinfo.io/account/token>.
3. Set on Railway:
   ```bash
   railway variables set IPINFO_TOKEN=<your_token> \
     --service dchub-backend --environment production
   ```
4. Railway auto-redeploys. Verify:
   ```bash
   curl -H "X-Admin-Key: $DCHUB_ADMIN_KEY" \
     "https://dchub.cloud/api/v1/admin/ip-enrich?ip=8.8.8.8"
   # Expect: {"ip":"8.8.8.8","provider":"ipinfo","org":"AS15169 Google LLC",
   #          "company":"Google","country":"US",...}
   ```
5. The `/visitor-intelligence` dashboard's top-anonymous-IPs panel will
   resolve to companies on the next page load (cached 24h per IP).
